Users

Streampunk has support for Users and Groups. The intent is that there will be several levels of user:

  • Superusers - full access to everything
  • Admins - full read and write access. Can use the Admin GUI.
  • Con committee, dept heads - can read and write everything through the normal pages.
  • Con staff - can read everything, but not change it.
  • Members - can read public things, but not private. Can't change anything. Can send email to programme participants.
  • Everyone else - Can read public data

So, long-term, there's potential for running your Streampunk server as a publicly-accessible site, where each member of the con has been allocated their own User account which allows them to send emails to each other, and people who have not yet signed up for the con can still read the programme info that's visible. But all that's in the future.

Default configuration

When you create a Streampunk installation, default data gets loaded, and you're prompted to create a superuser account. The setup does create some access permissions, but not the role-based ones described above. So when you look at the data through the normal pages, you won't have any permissions to see any private data (i.e. anything with visible set to False). None of the editing links will appear.

If you log in as your superuser account, then all the private data and edit links appear, because superusers get all permissions, automatically.

At this point, you should click on Set up all the group permissions on the Main page. (Note: if you've also installed the demo data that comes with Streampunk, then this has already been done for you.) This will install a bunch of additional permissions that Streampunk uses:

  • Read_public
  • Read_private
  • Edit_programme
  • Edit_kit
  • ...
and so on. These are the permissions that Streampunk uses to determine whether it's okay to display information, or whether to allow the current user to edit the data.

Once these permissions are set up, when you create new users through the Admin GUI, you can allocate appropriate permissions to them. You can also set up groups (con staff, programme staff, tech crew, etc.) and allocate the permissions to them, instead, which will make it easier when you create new users.
